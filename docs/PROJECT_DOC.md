# Ethereal Voyager — 特价机票发现平台

> 一个以「降低决策成本」为核心的机票信息聚合产品，聚合各大 OTA 平台和航司特价信息，通过价格透明化、规则可视化和智能推荐，帮助用户高效发现高性价比航班。

---

## 一、项目背景与痛点分析

### 1.1 行业现状

各大 OTA 平台（携程、飞猪、去哪儿）和航空公司频繁推出特价机票活动，但用户面临三大核心困境：

| 痛点 | 表现 | 影响 |
|:-----|:-----|:-----|
| **信息碎片化** | 特价信息散落在 10+ 个平台，需逐个比价 | 用户耗费大量时间，仍可能遗漏最优价格 |
| **规则不透明** | 退改签政策、行李额度等关键信息深藏在购票流程后期 | 用户购票后才发现限制，体验极差 |
| **决策疲劳** | 面对同一航线数十个价格选项，缺乏判断依据 | 用户犹豫不决，错过最佳购票时机 |

### 1.2 目标用户画像

- **价格敏感型旅客**：大学生、年轻白领，愿意为低价调整出行时间
- **频繁出行商旅**：需要快速决策，重视准点率和服务质量
- **灵感型旅行者**：没有固定目的地，被低价触发出行欲望

### 1.3 竞品不足

| 竞品 | 优势 | 不足 |
|:-----|:-----|:-----|
| 携程/飞猪 | 数据全，可直接购票 | 信息密度高，缺乏特价聚焦，规则展示滞后 |
| Google Flights | 价格趋势图优秀 | 国内覆盖弱，无中文优化 |
| 天巡 (Skyscanner) | 跨平台比价 | 跳转体验割裂，无规则前置 |

**本产品的差异化定位**：不做 OTA，做**决策辅助层**——聚合信息 → 前置规则 → 量化性价比 → 直达购票。

### 2.2 三页结构详解

#### 页面一：首页

首页承担**流量入口 + 价值展示**的双重职责：

**探索 Tab：**

| 模块 | 功能 | 设计意图 |
|:-----|:-----|:---------|
| Hero 搜索栏 | 出发城市 / 目的城市 / 出发日期 + 搜索按钮 | 降低使用门槛，一目了然的核心交互 |
| 分类标签 | 国内精选 / 国际大促 / 直飞特惠 / 本周末 / 说走就走 | 覆盖主流出行场景，减少搜索操作 |
| 特价卡片 | 城市图片 + 折扣 Badge + 价格 + 退改标签 + 剩余票量 | **规则前置**：不点进去就能看到关键限制条件 |
| 订阅 Banner | 邮箱订阅 CTA | 用户留存抓手 |

**设计亮点** — 每张特价卡片底部的「价格状态条」：
- 📉 当前价格处于近三月最低点 → 传递紧迫感
- 🔥 限时大促：今日仅剩 3 张 → FOMO 效应驱动转化

**我的监控 Tab：**

| 模块 | 功能 | 设计意图 |
|:-----|:-----|:---------|
| 监控卡片 | 目标价 vs 当前价 + 趋势迷你图 | 用户自定义「心理价位」，系统持续追踪 |
| 高亮提醒卡 | 接近目标价时自动高亮，蓝紫渐变强调 | 最醒目的视觉权重给「该出手了」的信号 |
| 统计看板 | 今日发现低价座位 / 合作品牌数 / 系统运行状态 | 建立平台可信度 |

#### 页面二：搜索结果页

搜索结果页承担**比较决策 + 快速筛选**的职责：

| 模块 | 功能 | 设计意图 |
|:-----|:-----|:---------|
| 价格趋势图 | 18日柱状图，当前日橙色高亮 + 低价日深色标注 | 回答「现在买划算吗？」 |
| 筛选面板 | 价格区间 / 航司 / 中转 / 行李 | 四维筛选覆盖核心决策因子 |
| 航班卡片 | 航司 + 时间 + 准点率 + 服务图标 + 性价比指数 | **量化决策**：9.8 分 > 主观判断 |
| 数据源标注 | "来自携程" / "来自飞猪" | 透明溯源，增强信任 |

**「性价比指数」设计逻辑**：综合价格、准点率、服务配置、舱位等多维度计算的 0-10 评分，帮用户从「全看价格」转变为「看综合性价比」。

#### 页面三：航班详情页

详情页承担**消除疑虑 + 促成转化**的职责：

| 模块 | 功能 | 设计意图 |
|:-----|:-----|:---------|
| 航班时间轴 | 出发/到达时间 + 航站楼 + 机型 + 航线可视化 | 全貌一览 |
| 规则透明化 | 行李额度 / 退改政策 / 机上服务 三卡片 | **核心差异化**：把导致投诉的信息前置 3 步 |
| 信任标签 | ⚡出票极速 / ✅官方认证 / 👁无隐形消费 | 降低心理门槛 |
| 价格明细 | 票价 + 税费 = 总额 | 拆解透明化，杜绝「结账才知道真实价格」 |
| 价格趋势 | 30天走势图 + 历史最低标注 | 回答「这个价格值不值？」 |
| 相似推荐 | 3条替代航线 | 挽留因价格/时间不满意的用户 |
| 目的地灵感 | 文案 + 旅行指南入口 | 情感化设计，提升转化率 |
| 预订按钮 | **直接跳转 OTA 购票页** | 最短路径完成转化 |

### 3.3 数据驱动设计

前端采用完全数据驱动的渲染模式，核心代码仅需关注一个配置项：

```javascript
const API_CONFIG = {
  BASE_URL: 'https://xxx.github.io/flight-api/data',
  USE_MOCK: true  // 后端就绪后改为 false
};
```

**渲染流程**：
1. 页面加载 → 调用 `getDeals()` / `getMonitors()` 等数据获取函数
2. 若 `USE_MOCK = true` → 直接返回内置 Mock 数据
3. 若 `USE_MOCK = false` → fetch 远端 JSON 文件
4. fetch 失败 → 自动降级到 Mock 数据（优雅降级）
5. 数据到手 → 调用对应 `render*()` 函数动态生成 DOM

**每张卡片都包含 `bookingUrl` 字段**，点击「选订航班」/「立即预订」→ 新标签页直接打开 OTA 购票页面。

### 3.4 API 接口设计

已设计 4 个标准化 JSON 接口，详见 [API_DOCS.md](./API_DOCS.md)：

| 接口 | 文件路径 | 用途 |
|:-----|:---------|:-----|
| 首页特价 | `/data/deals.json` | 3+ 张特价卡片数据 |
| 搜索结果 | `/data/search/{FROM}-{TO}.json` | 航班列表 + 价格趋势 |
| 航班详情 | `/data/flights/{ID}.json` | 完整航班信息 + 规则 |
| 监控列表 | `/data/monitors.json` | 用户订阅航线 |

后端只需输出符合格式的 JSON 文件并推送到 GitHub Pages，前端自动拉取渲染。

### 3.5 购票链接策略

系统支持多 OTA 平台的购票链接，格式示例：

```
# 携程
https://flights.ctrip.com/online/list/oneway-SHA-KIX?depdate=2026-04-20

# 飞猪
https://s.taobao.com/search?q=机票+上海+京都

# 去哪儿
https://flight.qunar.com/site/oneway_list.htm?searchDepartureAirport=上海&searchArrivalAirport=京都
```

每条航班数据的 `bookingUrl` 字段由后端爬虫根据数据来源自动填入，前端直接 `window.open()` 跳转。

## 五、核心产品亮点

### 5.1 规则透明化（解决「看不懂」）

传统 OTA 的退改签政策、行李限制深藏在购票流程的第 3-4 步，用户往往付款后才发现：
- 「不可退改」的特价票
- 「仅含 7kg 手提」的廉航舱位
- 隐藏的燃油附加费

**本产品的做法**：
- **首页卡片** → 直接展示「限改签」「不可退改」「直飞」「可退票」标签
- **详情页** → 独立「规则透明化」模块，行李/退改/服务三卡片并列
- **价格明细** → 票价 + 税费 = 总额，杜绝隐形加价

### 5.2 性价比指数（解决「选不准」）

面对同一航线的 5+ 个航班，价格差异可能只有几百元，但服务差异巨大。传统 OTA 依赖用户自行判断。

**本产品的做法**：引入 0-10 分的「性价比指数」，综合以下维度：
- 价格（相对历史均价的折扣幅度）
- 准点率（航班历史准点数据）
- 服务配置（餐食、行李、WiFi、充电）
- 舱位等级

### 5.3 价格趋势与预测（解决「不知道什么时候买」）

用户最常见的困惑：「现在这个价格是高还是低？该等还是该买？」

**本产品的做法**：
- **搜索页** → 18日价格趋势柱状图，当前日橙色高亮
- **详情页** → 30天走势 + 历史最低价标注
- **绿色文案提示** → "当前处于价格低位，建议立即预订"

### 5.4 降价订阅（解决「找不到」）

对于有明确出行计划但不急于购票的用户：
- 设定**目标价格**，系统持续监控
- 价格接近目标时**高亮提醒**
- 达到目标价后一键跳转购票

### 5.5 数据溯源（建立信任）

每条航班信息都标注来源平台（携程/飞猪/去哪儿），用户清楚知道：
- 这个价格来自哪个平台
- 点击「预订」会跳转到哪里
- 最终成交价以目标平台为准

## 六、后端架构方案

### 6.1 推荐方案：GitHub Actions + GitHub Pages

**零成本、零运维**的后端方案：

```yaml
# .github/workflows/crawl.yml
name: Crawl Flight Deals
on:
  schedule:
    - cron: '0 */2 * * *'  # 每2小时执行
  workflow_dispatch:        # 支持手动触发

jobs:
  crawl:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install requests beautifulsoup4
      - run: python scripts/crawl.py
      - run: |
          git config user.name "GitHub Actions"
          git config user.email "actions@github.com"
          git add data/
          git diff --staged --quiet || git commit -m "Update flight data"
          git push
```

**爬虫数据源建议**：

| 数据源 | 数据类型 | 爬取难度 | 反爬风险 |
|:-------|:---------|:---------|:---------|
| 携程特价页 | 特价航班列表 | ⭐⭐ | 中 |
| 飞猪促销页 | 限时特惠 | ⭐⭐ | 中 |
| 航司官网 | 会员专享价 | ⭐⭐⭐ | 低 |
| 天巡 API | 跨平台比价 | ⭐ | 低 (有公开 API) |

### 6.2 输出文件结构

```
flight-api/
├── data/
│   ├── deals.json          ← 首页特价（3-6条）
│   ├── monitors.json       ← 热门监控航线
│   ├── search/
│   │   ├── SHA-PEK.json    ← 上海→北京
│   │   ├── SHA-KIX.json    ← 上海→京都
│   │   ├── PEK-SYD.json    ← 北京→悉尼
│   │   └── ...
│   └── flights/
│       ├── flight_001.json
│       ├── flight_002.json
│       └── ...
├── scripts/
│   └── crawl.py            ← 爬虫主脚本
└── .github/
    └── workflows/
        └── crawl.yml       ← 定时任务配置
```

### 6.3 前后端对接

前端只需要修改一行代码即可完成对接：

```javascript
const API_CONFIG = {
  BASE_URL: 'https://yourusername.github.io/flight-api/data',
  USE_MOCK: false  // ← 改这里

```

## 十、技术亮点总结

面试/答辩时可重点强调的技术决策：

1. **单文件 SPA 架构**：HTML/CSS/JS 内联，零依赖零构建，任何静态托管平台一键部署
2. **数据驱动渲染**：所有 UI 从 JSON 数据动态生成，一行配置切换 Mock/真实数据
3. **优雅降级**：API 请求失败自动回退到内置 Mock 数据，保证页面永远可用
4. **购票链接直达**：每条数据包含 `bookingUrl`，用户点击直接跳转 OTA 完成购买
5. **CSS 设计系统**：CSS 变量驱动的完整 Design Token 系统，保证全局一致性
6. **前后端分离**：前端纯静态，后端纯 JSON，通过 GitHub Actions 实现零成本自动化
